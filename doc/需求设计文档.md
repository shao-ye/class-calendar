# 孩子每日课程记录日历 —— 需求设计文档

## 一、项目概述

一个以**日历形式**记录孩子每日所上课程的轻量级 Web 小工具。家长每天在日历对应日期上记录孩子上了什么课、上了多久、备注与照片，并能按周 / 月查看统计。

- **使用对象**：一期为家长本人（单用户、单个孩子）；二期开放多用户注册
- **核心场景**：每天花几秒钟点开日历某一天，选择课程、填时间段、写两句备注，必要时传张照片
- **部署方式**：Cloudflare 全家桶部署（仅用免费方案）
- **访问域名**：`class.shaoruiyi.com`（已上线）
- **项目根目录**：`class/`，需求文档位于 `class/doc/`

> **实现状态（截至上线）**：一期已开发完成并部署到 `https://class.shaoruiyi.com`。已落地：登录/会话、日历月/周视图、当日记录增删改、课程库与课时包、统计、照片上传(R2)、自动排课与计划课"确认到课"、法定假日同步（独立 Worker 定时 + 手动）。
> **已敲定的实现选择**：周起始日=**周一**；前端框架=**Vue 3 + Vite**（未引入 UI 组件库，日历与图表均自绘 CSS）；统计图表=**CSS 柱状**（未用 ECharts/Chart.js）；数据导出=暂未纳入一期。
> 配套文档：[项目架构文档](项目架构文档.md)、[开发设计文档](开发设计文档.md)、[部署说明文档](部署说明文档.md)。

## 二、需求确认结论

| 维度 | 确认结果 |
| --- | --- |
| 孩子数量 | 单个孩子（界面不做切换，数据模型预留扩展） |
| 访问控制 | 一期：单一用户名 + 密码登录；二期：多用户注册 |
| 微信登录 | 一期/二期**暂不实现**，但**架构需预留**，后期可平滑接入（详见 5.5） |
| 课程录入 | 下拉选择预设课程；下拉框可输入文字，未匹配到选项时**自动创建新课程**（类似腾讯在线表格下拉建值逻辑） |
| 附加功能 | 月度 / 周度统计、课程时长 / 时间段、备注、照片 |

## 三、分阶段实施规划

### 第一阶段（一期）—— 单用户 + 基础功能
- 单一用户名 + 密码登录
- 日历主视图（月 / 周）
- 当日课程记录的增删改（课程下拉建值、时间段 / 时长、状态、备注）
- 照片上传
- 周 / 月统计
- 课程库管理

### 第二阶段（二期）—— 多用户
- 用户注册 / 登录（用户名 + 密码）
- 数据按用户隔离（每个用户独立的课程库、记录、照片）
- 沿用一期全部功能，无需重写

### 远期可选 —— 微信登录
- 架构已预留（见 5.5），届时新增一种登录方式即可，**不需要改动核心表结构**
- 前置条件（务必知悉）：微信网站应用扫码登录需**企业主体** + 微信开放平台认证（约 300 元/年）+ 域名 **ICP 备案**；个人主体无法办理网站扫码登录，若坚持微信一键登录，更现实的路线是单独做**微信小程序**版本

## 四、功能需求

### 4.1 登录认证
- **一期**：单一用户名 + 密码登录
- **二期**：开放注册，支持多用户
- 密码在服务端以**哈希**形式存储与校验（建议 PBKDF2 / bcrypt），**前端永不接触明文哈希**
- 登录成功后下发签名 Cookie（有效期可设较长，如 30 天），实现"一次登录长期免登"
- 未登录访问任何页面 / 接口一律跳转登录页 / 返回 401

### 4.2 日历主视图（核心）
- 默认展示**当前月**的日历网格（周一 / 周日起始可配置）
- 每个日期格子显示当天已记录的课程：以**彩色实线标签**展示课程名（颜色取自课程库配色）
- 同时显示**计划课（虚线标签）**：根据课程排课规则（见 4.8）自动预排的未来上课日
- **法定假日**格子加"休"标记并淡色底；课程若设"休法定假日"则跳过假日不排
- 当天有多门课时叠加显示，超出则显示"+N"；图例区说明实线/虚线/假日含义
- 支持上 / 下月切换、回到今天
- 顶部提供"周视图 / 月视图"切换
- 点击某一天 → 打开当天**记录编辑面板**

### 4.3 当日记录编辑
- 一天可记录**多门课程**（一条课程记录 = 一节课）
- 单条记录字段：
  - **课程**：下拉选择 + 可输入创建（见 4.4）
  - **时间段**：开始时间、结束时间（自动算出时长）。**选中课程时自动带出该课程的默认时间段**（见 4.4），固定排课无需重复填写，偶尔调课再手动改即可；也可留空
  - **状态**：到课 / 请假 / 调课（默认"到课"）
  - **备注**：多行文本
  - **照片**：可上传 0~N 张
- 支持新增、编辑、删除单条课程记录

### 4.4 课程库与"两层创建"逻辑
课程的创建分两层，互不干扰：

**① 快速建值（记录课程时）**
- 下拉展示全部已有课程，支持**输入关键字过滤**
- 当输入内容**未匹配到任何已有课程**时，下拉底部出现"➕ 创建『xxx』"，点击即时建好并选中
- 快速创建只录**名字**（自动配色、无默认时间、默认不统计课时），保证记录当下不被打断

**② 课程编辑（课程库管理页）**
- 点课程进入编辑，配置：名称、颜色、**默认时间段 / 默认时长**、**上课星期 / 是否休法定假日**（见 4.8）、**是否统计课时**、**课时包**（见 4.7）
- 可对快速创建的课程补全上述配置
- 支持改名、改色、停用 / 删除（删除时提示该课程已有 N 条历史记录）
- 创建后可弹轻量"是否现在完善时间 / 课时"引导（可跳过）

**默认时间段**：兴趣班多为固定排课，给每门课配默认开始时间与时长；记录时选中课程即自动带出，免去每次重复填写（核心减负设计）。

### 4.5 统计分析
- **时间范围**：本周 / 本月 / 自定义区间
- **统计维度**：
  - 各课程出现次数（柱状或排行）
  - 各课程累计时长
  - 总上课天数 / 总课时数
  - 出勤情况（到课 / 请假 / 调课占比）
  - **花费统计**（基于课时包金额）：累计花费、各学科花费、平均课时费、按购买日期落入选定区间的时段花费
- 以图表 + 列表两种形式呈现

### 4.6 照片管理
- 单条记录可附照片，缩略图展示，点击查看大图
- 照片上传前前端做压缩（控制体积，节省存储与流量）
- 照片与课程记录绑定，删除记录时一并清理

### 4.7 课时管理（课外班课时包，循环统计）
针对课外补习班"按课时包购买、要看剩余、用完续费"的场景。

- **按课程开关**：每门课可设"是否统计课时"。课外补习班开启，学校课 / 兴趣课关闭（关闭则不显示任何课时信息）
- **课时包**：开启后录入课时包 = 购买总课时 + 购买日期 + **金额**（购买日期与金额均可编辑），自动算出**平均课时费**（金额 ÷ 总课时）
- **扣减规则**：每上一节默认扣 **1 课时**，可在该条记录手改为 1.5 / 2（长课）；**仅"到课"扣减，请假 / 调课不扣**
- **手动校正已上课时**：课时包的"已上课时"与"总课时"均可手动改，覆盖两类场景——① **中途接入系统**（如已上 10 节才开始用，新增包时直接把已上填 10）；② **补漏 / 记错**（漏记或多记一节，直接调整已上数）。校正与日常按记录扣减并行，互不冲突
- **直观展示**：课程库卡片显示进度条 + "剩 12/20 课时"；剩余 ≤ 阈值（如 3）变红提醒"快用完了"
- **循环统计**：同一门课的多个课时包串成一条**时间线**——同时只有一个"生效中"的包参与扣减；用完后点"续费 / 新增课时包"，新包计数归零重新算，**旧包归档为"已用完"并保留在时间线**，可回看一共买过几期、每期上了多少
- 当前剩余 = 生效包总课时 − 生效包已用

### 4.8 排课规则与自动预排
让固定排课的课程在日历上自动"长出"未来的上课日，减少手动安排。

- **上课星期**：每门课可设一个或多个星期几（如英语=周二、周四），配合默认时间段（4.4）形成排课规则
- **自动预排**：从今天起，按"上课星期"向后投影出未来的上课日，在日历以**计划课（虚线标签）**显示
  - 统计课时的课：预排数量 = **剩余课时**（剩多少节就排多少节，排完为止）
  - 不统计课时的课：在可视范围（如当月/未来若干周）内按星期循环显示
  - 当天若已有该课程的**实际记录**，则不再重复显示其计划课
  - **计划课"确认到课"**：点开当天，计划课会列在抽屉里，提供「确认到课」一键转为真实记录（套用默认时间、到课扣 1 课时），或「编辑」后再保存；月、周视图均显示计划课
  - 计划投影在前端完成（读 `holidays` + 课程排课规则 + 剩余课时），不写库
- **是否休法定假日**（每门课开关）：
  - 开：遇法定假日**跳过不排**，顺延到下一个符合星期的日子
  - 关：假日**照常排课**（如游泳馆假日不休）
- **生效范围（重要）**：排课规则与假日设置**仅影响今天及以后**的预排。**历史已记录的课时不受任何影响**——无论改上课星期、改休假日开关，还是同步到新的假日数据，都不会回溯改动过去已发生的记录与课时扣减
- **法定假日数据**：来自**外部接口**，由后端定时同步进 `holidays` 表，前端只读（详见 5.7）；支持手动同步与手动校正

## 五、技术方案（Cloudflare 部署）

### 5.1 架构选型

| 层 | 选型 | 说明 |
| --- | --- | --- |
| 前端 | Cloudflare Pages | 静态站点托管，自带 CDN 与 HTTPS |
| 后端 API | Pages Functions（基于 Workers） | 与前端同仓部署，免单独维护 Worker |
| 数据库 | Cloudflare D1（SQLite） | 存用户、课程库、上课记录、照片元数据 |
| 图片存储 | Cloudflare R2 | 存照片二进制文件，元数据落 D1 |
| 密钥 | Workers Secret / 环境变量 | 存 Cookie 签名密钥等 |

> 前端框架建议 **Vue 3 + Vite** 或 **React + Vite**，UI 可选 Element Plus / Naive UI（下拉建值组件成熟）。日历可用 FullCalendar 或自绘网格。统计图表建议 ECharts 或 Chart.js（轻量）。

### 5.2 为什么免费方案就够（选型论证）

记课程是典型的"低频写、低频读"场景，离免费额度天花板很远；即便二期多用户也很难触顶。

| 资源 | 免费额度 | 本应用用量预估 | 结论 |
| --- | --- | --- | --- |
| Pages 托管 | 请求不限量，500 构建/月 | 偶尔发版 | 绰绰有余 |
| Workers/Functions | 10 万请求/天 | 每天几十~几百次 | 用不到 1% |
| D1 | 5GB、500 万行读/天、10 万行写/天 | 每年几千行 | 多用户也撑很久 |
| R2 | 10GB 存储、**出口流量免费** | 压缩后每张几百 KB | 够用多年 |

> 结论：一期、二期均无需 VPS。仅当未来出现"长驻后台进程、单请求重计算、需要传统关系库复杂事务"等重型场景才考虑 VPS。

### 5.3 子域名配置
1. Cloudflare 控制台 → `shaoruiyi.com` → DNS，子域名通过 Pages 自定义域名自动绑定 CNAME
2. 在 Pages 项目的 **Custom domains** 添加 `class.shaoruiyi.com`，Cloudflare 自动签发证书

### 5.4 数据模型（D1 / SQLite）

设计原则：**一期就按多用户结构建表**（所有业务表带 `user_id`），二期开放注册时无需改表 / 迁移数据；认证方式独立成表，为微信登录预留。

```sql
-- 用户表
CREATE TABLE users (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  username      TEXT NOT NULL UNIQUE,        -- 登录用户名
  password_hash TEXT,                        -- 密码哈希（微信用户可为空）
  display_name  TEXT,                         -- 昵称
  avatar_url    TEXT,                         -- 头像
  created_at    TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 认证身份表（一个用户可绑定多种登录方式，为微信预留）
-- provider: 'password' | 'wechat' | 'wechat_mp' ...
-- provider_uid: password 方式存 username；wechat 方式存 openid/unionid
CREATE TABLE auth_identities (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id       INTEGER NOT NULL,
  provider      TEXT NOT NULL,
  provider_uid  TEXT NOT NULL,
  union_id      TEXT,                         -- 微信 unionid，预留
  created_at    TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY (user_id) REFERENCES users(id),
  UNIQUE (provider, provider_uid)
);

-- 课程库（按用户隔离）
CREATE TABLE courses (
  id               INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id          INTEGER NOT NULL,
  name             TEXT NOT NULL,
  color            TEXT NOT NULL DEFAULT '#3b82f6',
  default_start    TEXT,                       -- 默认开始时间 HH:MM，可空
  default_duration INTEGER DEFAULT 60,         -- 默认时长(分钟)
  weekdays         TEXT,                        -- 上课星期，如 "2,4"（1=周一…7=周日），用于自动预排
  skip_holiday     INTEGER NOT NULL DEFAULT 1, -- 遇法定假日是否跳过排课
  track_hours      INTEGER NOT NULL DEFAULT 0, -- 是否统计课时（课外班=1）
  is_active        INTEGER NOT NULL DEFAULT 1,
  created_at       TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY (user_id) REFERENCES users(id),
  UNIQUE (user_id, name)                       -- 同一用户内课程名唯一
);

-- 课时包（循环统计：一门课可有多个包，同时仅一个 active）
CREATE TABLE course_packages (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  course_id     INTEGER NOT NULL,
  total_hours   REAL NOT NULL,                 -- 购买总课时
  opening_used  REAL NOT NULL DEFAULT 0,       -- 期初已用（中途接入/手动校正的基线）
  purchase_date TEXT,                          -- 购买日期 YYYY-MM-DD
  amount        REAL,                          -- 金额（可选）
  note          TEXT,
  status        TEXT NOT NULL DEFAULT 'active', -- active 生效 / done 已用完
  created_at    TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY (course_id) REFERENCES courses(id)
);
-- 当前已用 = opening_used + SUM(records.consumed_hours WHERE package_id 指向本包且 status='attended')
-- 当前剩余 = total_hours - 当前已用
-- 说明：opening_used 承载“中途接入/手动校正”，日常按记录自动累加，二者并行

-- 法定假日表（全局共享，非按用户隔离；供自动排课跳过假日用）
CREATE TABLE holidays (
  date        TEXT PRIMARY KEY,                -- YYYY-MM-DD
  name        TEXT,                            -- 假日名称
  is_workday  INTEGER NOT NULL DEFAULT 0       -- 1=调休补班日（虽是周末但要上课）
);

-- 上课记录（一条 = 一节课，按用户隔离）
CREATE TABLE records (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id     INTEGER NOT NULL,
  date        TEXT NOT NULL,                  -- YYYY-MM-DD
  course_id   INTEGER NOT NULL,
  start_time  TEXT,                           -- HH:MM，可空
  end_time    TEXT,                           -- HH:MM，可空
  duration    INTEGER,                        -- 时长(分钟)
  status      TEXT NOT NULL DEFAULT 'attended', -- attended/leave/reschedule
  package_id  INTEGER,                         -- 扣减的课时包（统计课时的课程才有）
  consumed_hours REAL DEFAULT 1,               -- 本节扣减课时，默认1，可改1.5/2
  note        TEXT,
  created_at  TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at  TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY (user_id) REFERENCES users(id),
  FOREIGN KEY (course_id) REFERENCES courses(id),
  FOREIGN KEY (package_id) REFERENCES course_packages(id)
);
CREATE INDEX idx_records_user_date ON records(user_id, date);

-- 照片（元数据，文件存 R2）
CREATE TABLE photos (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  record_id   INTEGER NOT NULL,
  r2_key      TEXT NOT NULL,                  -- R2 对象 key
  created_at  TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY (record_id) REFERENCES records(id)
);
```

> - 一期：`users` 表只有一条记录，`auth_identities` 只有一条 `provider='password'`；业务表 `user_id` 全部指向该用户。
> - 二期：开放注册，往 `users`/`auth_identities` 加行即可。
> - 多孩子扩展：未来加 `children` 表并在 `records` 加 `child_id`。

### 5.5 微信登录预留设计（关键）

为保证"以后想升级微信登录时架构能支持"，本期做如下预留，避免届时大改：

1. **登录方式与用户解耦**：用户身份（`users`）与登录凭据（`auth_identities`）分两张表。新增微信登录 = 在 `auth_identities` 插入一条 `provider='wechat'` 的记录并关联到某个 `user_id`，**core 表不动**。
2. **`union_id` 字段预留**：便于将来同一微信主体在多端（网站/小程序/公众号）打通。
3. **会话层抽象**：后端登录成功后统一签发同一种会话 Cookie / JWT，业务接口只认"当前 user_id"，不关心用户是用密码还是微信登录进来的。
4. **认证服务模块化**：后端把"校验凭据 → 找到/创建 user → 签发会话"抽象成统一流程，密码登录是其中一个 provider 实现，微信登录将来作为另一个 provider 接入。

### 5.6 接口设计（草案）

| 方法 | 路径 | 说明 | 阶段 |
| --- | --- | --- | --- |
| POST | `/api/auth/login` | 用户名 + 密码登录，下发会话 Cookie | 一期 |
| POST | `/api/auth/register` | 用户注册 | 二期 |
| POST | `/api/auth/logout` | 退出登录 | 一期 |
| GET | `/api/auth/me` | 取当前登录用户信息 | 一期 |
| GET | `/api/records?month=YYYY-MM` | 取某月全部记录 | 一期 |
| GET | `/api/records?date=YYYY-MM-DD` | 取某天记录 | 一期 |
| POST | `/api/records` | 新增一条课程记录 | 一期 |
| PUT | `/api/records/:id` | 修改记录 | 一期 |

所有 `/api/*`（除登录 / 注册 / 微信回调）均校验会话，并按 `user_id` 隔离数据（`holidays` 为全局共享，只读不分用户）。

### 5.7 法定假日数据对接（外部接口）

- **数据来源**：开源维护的中国节假日年度数据 holiday-cn，默认走 CDN 镜像 `https://cdn.jsdelivr.net/gh/NateScarlet/holiday-cn@master/{year}.json`（字段含 `date` / `name` / `isOffDay`），可用环境变量 `HOLIDAY_API_URL` 换源
- **对接方式（关键：前端不直连外部）**：
  1. **独立 Worker `class-holidays-sync`**（绑定同一个 D1）通过 **Cron Triggers** 定时拉取——因 **Pages Functions 不支持 Cron**，定时任务单独成一个 Worker
  2. 解析后 **以 `date` 为主键 upsert 进 D1 `holidays` 表**（`isOffDay=true` → 放假 `is_workday=0`；`isOffDay=false` → 调休补班 `is_workday=1`）
  3. 前端与排课逻辑只读 D1（`GET /api/holidays?year=`），**不直接访问外部 API**（规避 CORS、外部不稳定、限流）
- **两条同步触发途径**：
  - 自动：Worker Cron（当前 `0 18 28 * *`，每月一次，同步当年+次年）
  - 手动：「我的」页「同步法定假日」按钮 → `POST /api/holidays/sync`（即时补充）
- **兜底与幂等**：同步失败保留旧数据不清空；以 `date` upsert，重复同步不产生脏数据
- 详见 [部署说明文档](部署说明文档.md) 与 [开发设计文档](开发设计文档.md)
